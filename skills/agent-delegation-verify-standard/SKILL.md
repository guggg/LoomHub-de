## 用途 / What

當工作交給 AI subagent 執行（寫程式、改文件、大規模收斂），協調者（人或另一個 agent）必須
知道「這個結果可以信任嗎」，而不是單純相信 subagent 回報的「已完成」。這套標準定義：委派
實作與委派審查該用什麼角色、審查者該不該共享實作者的記憶、協調者親自驗證到什麼程度才能算
「驗收通過」。

一份符合這套標準的委派流程：
- 實作永遠交給有明確職責的實作角色（例如 senior-fullstack-engineer），不是協調者自己動手寫。
- 審查永遠開一個獨立、不共享實作者記憶的審查角色（例如 ruthless-code-reviewer），保持懷疑
  視角，不會被實作者的自我敘述牽著走。
- 協調者親自跑測試、親自抽查關鍵檔案、親自執行指令看真實輸出，而不是把 subagent 的文字報告
  當成事實。
- 只有審查通過 + 協調者親自驗證過，才回報「完成」或詢問下一步（例如 commit）。

持續遵守這套標準，能避免兩類常見失敗：（1）subagent 自我敘述「已完成」但實際留下斷裂引用
或邊界 bug；（2）審查者若與實作者共享記憶／假設，會傾向確認實作者的判斷而非獨立檢驗。

## 使用場景 / When

- 委派 subagent 修改程式碼、大規模改文件、或執行涉及刪除/搬移檔案的收斂工作，且結果會被
  拿去做後續動作（commit、push、對外發布）。
- 協調者本身無法（或不該）逐行手寫全部程式碼，但仍需要對最終結果負最終責任。
- 一次任務牽涉多個檔案、有可能因為改動範圍大而遺留死連結、斷裂引用、或邊界情況未覆蓋。

**不適用**：
- 單一、範圍極小、可一次讀完全部上下文並立即驗證結果的操作（例如改一行文字），直接動手比走
  完整委派流程更有效率。
- 純粹的資訊查詢或分析任務，沒有「結果需要被信任才能繼續」的下一步。

## 使用方式 / How

協調者在委派任務前後對照本文件的 §適用原則 / 各階段標準 自我檢查；審查一份委派流程是否
合格時，逐維度核對，指出不符合的具體項目。

## 前置條件 / Prerequisites

- 有明確定義、職責分工清楚的 subagent 角色可用（至少一個實作角色、一個獨立審查角色）。
- 協調者本身有能力執行驗證步驟（跑測試、讀檔案、執行指令），而不是只能被動接收報告。

## 適用原則 / 各階段標準

| 維度 | 原則（為什麼重要） | 標準（怎樣算做到） | 怎麼判斷 |
|---|---|---|---|
| 實作與審查角色分離 | 同一個角色同時寫程式又審查自己的程式，容易對自己的假設視而不見 | 實作交給有明確職責的實作角色；審查交給另一個獨立角色，不是同一個 agent 自我審查 | 檢查委派紀錄：實作與審查是否為不同的 subagent 呼叫，而非同一個 agent 說「我審查過了」 |
| 審查者獨立性（不共享記憶） | 若審查者延續實作者已有的上下文/假設，容易傾向確認而非挑戰 | 審查者以全新、不帶實作者記憶的 agent 執行，即使這會犧牲一些效率 | 檢查審查任務是否重新提供完整背景脈絡給一個新 agent，而非用 SendMessage 接續實作者本人 |
| 協調者親自驗證 | Subagent 的文字報告只是「它認為自己做了什麼」，不是事實本身 | 協調者親自執行至少一項可驗證的動作（跑測試、跑建置腳本、grep 關鍵字、讀取關鍵檔案內容），且結果與報告一致 | 檢查協調者的回報是否包含「我自己跑了 X，結果是 Y」，而非只複述 subagent 的敘述 |
| 全面性檢查（不只抽查亮點） | 大範圍改動最容易在「沒被特別提到的角落」留下斷裂引用或遺漏 | 涉及刪除、搬移、合併檔案的任務，驗證時要包含全 repo 搜尋殘留引用，而非只檢查被明確要求的幾個檔案 | 檢查是否執行過 `grep -rn` 之類的全域搜尋確認無死連結，而非只讀了被指名的檔案 |
| 委派同一批任務時接續而非重開 | 對同一批相關檔案的後續修改任務開新 agent，等於讓它重新繳一次「認識這個 repo」的稅，且可能與前次改動的假設脫節 | 若後續任務與前一個實作 agent 剛完成的工作高度相關，優先用 SendMessage 接續它；只有任務性質真正無關、或審查任務（見上一維度）才開新 agent | 檢查是否有「本可接續卻重新開新 agent 處理同一批檔案」的情況，若有即為違反 |
| 只有驗證通過才問下一步 | 過早詢問「可以 commit 了嗎」等於把「有沒有做對」的判斷責任推給使用者 | 只有審查通過、且協調者親自驗證過（見上方三項）才回報完成或詢問是否要進行下一步動作 | 檢查回報「完成」或詢問 commit 的那一刻，前面是否已經有審查結論與親自驗證的證據，而非審查一啟動就急著問 |

## Demo / 範例

```demo-conversation
user: 幫我把這 5 個檔案的過時範例名稱換成真實存在的資產名，改完 commit。

agent: （委派 senior-fullstack-engineer 執行替換，並要求列出所有修改處、跑測試、但不要
commit/push）

agent: （實作完成後，開一個全新的 ruthless-code-reviewer——不接續實作者的 agent、不共享
它的假設——要求逐一核對每處引用是否真的替換成真實存在的資產，並實際跑一次相關指令驗證）

agent: （審查回報「有一處遺漏：docs/foo.md 第 12 行仍指向舊名稱」）

agent: 我自己讀了 docs/foo.md 確認這個遺漏是真的，修正後重新跑一次全 repo grep 確認沒有
殘留，再跑測試確認全過。

result: 三項驗證（審查發現的遺漏已修正、grep 無殘留、測試全過）都通過後，才回報：「已完成，
要 commit 嗎？」
```

## 取用 / 套用

這是一份**協作方法論**，沒有可安裝的檔案——直接參照或複製本文的原則與各階段標準：

1. 委派實作前，先確認有沒有明確分工的實作角色可用；委派審查時，優先開一個獨立、不共享實作
   者記憶的新角色。
2. 實作完成後，不要自己動手改，先交給獨立審查者。
3. 審查完成後，協調者親自執行至少一項可驗證的動作（跑測試、跑建置、grep、讀檔案），確認
   結果與報告一致，且涵蓋全面搜尋而非只抽查亮點。
4. 只有審查通過且親自驗證過，才回報「完成」或詢問下一步（例如是否 commit）。
5. 若後續任務與前一個實作角色剛完成的工作高度相關，優先接續它而非重新開一個 agent；審查
   角色則維持每次獨立、不接續。

這份標準本身不會被「執行」——它是你判斷「這次委派 subagent 的流程做得夠不夠嚴謹」時參照
的基準。
