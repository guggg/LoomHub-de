# ADR-0006：檢核與重疊偵測由本機 AI Agent 依 AGENTS.md 執行

> 狀態：Accepted ｜ 日期：2026-07-09
> 相關：[01-prd](../01-prd.md) FR-2 / FR-3

## Context

hub 需要品質治理：規範一致（frontmatter 齊備、白名單、semver、人看段落）與去重（新 skill 是否與既有重疊）。約束與取向（見 PRD）：

- **分享 / 更新要極簡、不造成成員麻煩**——所有 skill 更新頻率相近，團隊採**信任制**，**不強制開 PR、不設伺服器端 CI 擋關**。
- 品質靠**貢獻者自己的 AI Agent 在本機依 `AGENTS.md` 檢核**，以 prompt + 明確步驟定義、可一致重跑。
- 規範性問題為**提交前的硬性自我把關**（AI 檢核未過就不該提交）；重疊判斷為**輔助**（貢獻者參考後自行決定）。
- 起步階段完全信任，push 不設核准閘門；未來團隊變大可再加（見 ADR-0007）。
- 註：「不設 CI」專指**不設檢核擋關**，不禁止**自動部署管線**（目錄頁 push 後自動重建仍可，見 Spec §7.1）。

## Decision

- repo 根目錄放一份 **`AGENTS.md`**（跨廠商指令慣例，三家 agent 皆可讀），內含「提交前檢核清單」：
  1. 規範檢核步驟（比對 `skill.schema.json`：必填欄位、type/category 白名單、semver 格式、必要人看段落存在）——**任一不過即應修正後再提交**。
  2. 重疊偵測步驟（**分類優先**：先判斷新 skill 的 category，再只掃該 category 內既有 skill 的 description 比對，見下）——輸出重疊判斷 + 建議。
  3. 產出人類可讀檢核結論供貢獻者自檢。
- **貢獻流程極簡（免 PR）**：通過本機 AI 檢核後，**直接 commit + push 到主線**即可分享 / 更新，不強制 PR / reviewer。信任成員會依 `AGENTS.md` 自檢。
- 檢核所需的機器可讀規範集中於 repo 內 `skill.schema.json`（與目錄產生共用）。

### 重疊偵測：分類優先（省時省 token）

不逐一掃全部 skill 的 description（雖最保險但慢且耗 token）。改為兩步：

1. AI Agent 依 §4.2 的**通用分類規則**先判斷新 skill 的 `category`（穩定分類標準使不同 agent 歸類一致）。
2. **只掃該 category 內既有 skill 的 description** 判斷是否重疊。

重疊處理原則：**多數情況允許並存**，但貢獻者需在新 skill 的說明中**明確寫出與相似 skill 的區別 / 場景 / 應用差異**——AI Agent 偵測到相似時，主動要求貢獻者補上這段差異說明。

## Consequences

**正面**
- 分享 / 更新極簡：自檢過即 push，無 PR 摩擦，符合「不造成麻煩」的取向。
- 無需長駐 CI；符合低維運目標。
- 分類優先的重疊偵測大幅省時省 token，且靠統一分類規則提高歸類一致性。
- 檢核邏輯以 prompt + schema 明文化，任何成員 / 任何家 agent 可一致重跑。

**負面 / 取捨**
- 品質全靠信任 + 自檢，無機器強制——有人略過自檢就會漏。緩解：`AGENTS.md` 清楚、Loom 起草即帶檢核、`build-index` 產目錄時再跑一次規範檢核印警告（見 ADR-0004 / spec §7.1）。
- 分類優先可能漏掉「被歸到別類、但其實重疊」的 skill；接受此風險（換取效率），且並存本就多半允許。
- 未來若團隊變大、信任成本上升，可再加 PR / 輕量 CI，與本決策不衝突。

## Alternatives considered

- **強制 PR + reviewer 把關**：品質更穩，但增加分享摩擦，違背「更新要極簡」取向。→ 否決（起步）。
- **伺服器端 CI 硬性擋關**：最可靠但增運維、且與信任制取向不符。→ 降為未來可選保險層。
- **重疊偵測掃全部 skill**：最保險不漏，但慢且耗 token。→ 否決，改分類優先。
