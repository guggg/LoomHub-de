## 第一階段：完整開通流程（會動到真實全域檔）

**在任何資料夾**（不必是 LoomHub-de 專案內——這正是要測的重點：跨專案零摩擦）開一個新 session，貼：

```
我想裝 LoomHub-de，這是我們團隊的 AI Agent 資產分享中心。
repo: https://github.com/guggg/LoomHub-de
請照這個 repo 的 README「給 AI Agent 的安裝指示」幫我裝好。
```

**驗收重點（按流程順序）：**

1. **它會不會先問你 clone 路徑？**
   （設計要求：不寫死預設，agent 要先問使用者要 clone 到哪。若它自己選了路徑沒問你，記一筆——這是設計要求的行為，沒問就是沒照做。）

2. **它會不會正確跑 `install.sh`？**
   （應該看到 clone 成功、`npm install`、然後呼叫 `install-skill.mjs loom` 把 Loom 裝進你的 agent skill 目錄。）

3. **它會不會問你「你用哪家 agent」？**
   （Claude Code / Codex / Gemini，可多選。）

4. **⚠️ 檢查點——它要改你的 `~/.claude/CLAUDE.md` 前，會不會先說明用途、徵求同意？**
   設計要求它要講類似：「我想在你的 `~/.claude/CLAUDE.md` 加一段，讓我之後任何專案的 session 都會主動留意『這件事能不能變成 LoomHub-de 資產』，可以嗎？」
   **若它沒說明用途、只是默默要動手或隨口問一句「可以嗎」——先阻止它,回報這是個問題,再決定要不要放行。**
   同意後，可以自己核對 `~/.claude/CLAUDE.md` 有沒有出現 `<!-- LoomHub-de:start -->` 到 `<!-- LoomHub-de:end -->` 的區塊，內容是否跟 repo 裡 README/spec 定義的一致。

5. **它會不會做 onboarding 介紹？**
   （裝完後，應該會主動跟你介紹這個 hub 是什麼、怎麼瀏覽/安裝資產、怎麼分享、去哪看撰寫指引。）

**測完清理（若想恢復原狀）：**
```bash
# 還原全域檔（若步驟 4 有真的改動）
cp ~/.claude/CLAUDE.md.backup-before-loomhub-test-* ~/.claude/CLAUDE.md

# 移除裝好的 Loom（依它安裝時報的路徑調整）
rm -f ~/.claude/skills/loom ~/.agents/skills/loom
```

## 第二階段：主動偵測驗收（跨專案、免手動觸發）

若第一階段成功、你的 `~/.claude/CLAUDE.md` 已有偵測區塊，**開一個全新 session，在任何其他專案資料夾裡**，直接把下面情境當作你在跟它閒聊工作，**不要主動要求它分類**——看它會不會自己冒出來問。

### 情境 A（對應：skill）

> 「我最近每次接到資料異常的告警,都要先跑一串固定的 SQL 去 warehouse 撈:先看那張表最近的載入時間、再比對上游來源的筆數、再看有沒有重複 key。我把這串查詢跟判讀順序整理成一段給 AI 的指示,以後想直接叫它幫我做這個初步排查,不想每次自己貼。」

**驗收：它會不會主動問「這個要不要沈澱成 LoomHub-de 資產？」**（而不是你要求它才做）。若沒有主動問，代表全域偵測沒生效或訊號不夠明顯——記下來，回頭調整全域區塊的偵測描述。

若它主動問了、你答應了，接著看：它會不會呼叫 Loom（讀 `skills/loom/SKILL.md`）、判斷 type、起草、跑檢核。

## 第三階段：其餘 3 種產製（可用手動觸發，較快）

若第二階段驗證過主動偵測有效，其餘 3 個可直接手動觸發（不必再開新 session 硬等偵測），節省時間：

### 情境 B（對應：prompt）

> 「我常常要把一段落落長的英文技術文件濃縮成中文摘要給主管看,格式固定:三句話總結、關鍵風險、建議動作。我調了一個蠻好用的 prompt 範本,裡面有幾個地方要填(文件內容、受眾、想強調什麼)。想分享給團隊,大家自己複製去填就好。
> **我想把這個分享到 LoomHub-de,幫我判斷分類、起草、檢核。**」

### 情境 C（對應：mcp-server）

> 「我把我們內部那台 Athena（放 log 的）包成一個可以讓 agent 直接查的服務——設定好連線、限制成唯讀、只開放 query 跟 list-tables 兩個操作。裝好之後 agent 就多了『查 log』的能力,不用我手動貼 SQL 結果。連線設定跟金鑰要另外給。
> **我想把這個分享到 LoomHub-de,幫我判斷分類、起草、檢核。**」

注意：這個情境涉及 secret（連線金鑰）——**驗收重點是它會不會提醒用 env placeholder、不要寫死真實金鑰**（見 `docs/authoring/mcp-server-guide.md` 的安全段落）。

### 情境 D（對應：workflow）

> 「我們 code review 一直沒有統一標準,每個人抓的重點不一樣——有人只看邏輯對不對,有人會抓 commit message 寫得爛,有人完全不管測試覆蓋。我自己心裡有一套標準:改動的邏輯一定要有測試(能證明改了什麼、不是隨便補一個)、commit message 要講為什麼改而不是重複 diff 內容、不能留會偷偷改變行為的 TODO、PR 大小要能一次看完(不要把重構跟功能改動混在一起)。這不是一個工具或腳本,單純是我審 PR 時心裡對照的標準,也想讓自己寫 PR 前先自我檢查一輪。想把這套標準寫下來分享給團隊,讓大家審查跟自己寫 PR 時都能對齊同一套標準。
> **我想把這個分享到 LoomHub-de,幫我判斷分類、起草、檢核。**」

注意：驗收重點是它會不會正確判斷成 `workflow`而非 `skill`——這套東西**不會被執行去產出結果**,只是被人拿來自我檢查/審查時對照的標準（即使個人使用、只有一個人,也不影響分類——見 `docs/authoring/workflow-guide.md` §1 的判斷準則）。另外驗收末節是否正確為「取用 / 套用」（workflow 永遠是取用型,不會是「安裝」）。
